# v0.1.x — Quality-of-Life (Roll-overs, CSV Profile Model, Tray Quick-Capture)

**Date:** 2026-07-01
**Status:** Design — pending implementation plan
**Scope:** Two opportunistic patch releases (`v0.1.1`, `v0.1.2`) landing before the v0.2 web build. These are not on the existing roadmap (v0.2 web → v0.3 sync → …); they are cheap, day-to-day-value wins inserted around it.

## Motivation

The project foundation (testing, data integrity, architecture) is solid at v0.1.0. The next pressure is **daily-use stickiness / reduced input friction** — no hard deadline, but work should make the app more pleasant to actually use. The existing roadmap remains the plan; these items are opportunistic extras that do not displace v0.2.

Three deliverables, split along a natural fault line (in-app logic vs. OS integration):

| Release | Deliverable | Kind |
|---|---|---|
| **v0.1.1** | Budget roll-overs | Pure frontend/SQL, derived calc |
| **v0.1.1** | CSV import mapping-profile data model | Storage shape only (no UI) |
| **v0.1.2** | Tray quick-capture window | Tauri/Rust + new window + global shortcut |

## Out of scope (explicitly deferred)

- **FTS5 / full-text search.** LIKE queries are adequate for realistic personal-finance scale (hundreds to low-thousands of transactions per user). **Revisit only when search latency is measured on real data — not beforehand.** Not scheduled to a version; a maybe-never feature gated on a performance signal (and it would complicate v0.3/v0.4 sync, since FTS virtual tables don't merge across devices the way plain rows do).
- **Sync / E2EE.** Roadmap v0.3–v0.4. Schema decisions in this spec are verified sync-safe (see each section) but no sync work is included. Non-blocking note: research the multi-writer merge problem *before* the schema ossifies, even though implementation is far off.
- **CSV importer UI, parser, duplicate detection, column auto-detection.** All v0.3. This spec locks only the storage shape.
- **Web-build quick-add UI.** Tray/global-shortcut are desktop-only; the web (v0.2) will get its own in-page quick-add (e.g., `/` focus shortcut). Designed when v0.2 is concrete.

## Release v0.1.1a — Budget Roll-overs

### Concept

Unspent money in a category carries into the next month's available budget; overspent categories reduce it (envelope budgeting convention). **Pure derived value — no new stored columns for the calculation itself.** The one schema change is a per-category toggle.

### Formula

Roll-over is the cumulative prior surplus/deficit for a category, summed only over months that **have a budget row** for that category (YNAB-style gating):

```
rolledOver(typeId, month N) = Σ over months M < N where a budgets row exists for (typeId, M)
                              of [allocated(M) − spent(M)]
```

**Why budget-row-gated:** a category with early spending but no early budget allocation must not accumulate a phantom perpetual negative. Spending before the first budget row is treated as unbudgeted history and ignored for roll-over. Roll-over begins accruing only from the first month the user actually allocated to the category.

`spent(M)` reuses the existing `getSpentForBucket` SQL semantics: `SUM(expense.amount) − SUM(refund.amount)` for that type/month, income and transfers excluded. Because the SQL already nets out refunds, "lifetime spent" is well-defined.

The unrolled (cumulative) form is preferred over a month-by-month chain: no recursion, no chaining, no snapshot, one SQL query per category. Recalculating when a past transaction is edited is **correct behavior** — the money really did carry forward; this matches YNAB and envelope-budgeting expectations. Historical-report freezing is a v0.5 reporting concern, out of scope here.

### Per-category toggle

Most categories benefit from roll-over (car repair, gifts — sinking funds). A few want reset (groceries, dining). The escape hatch is **per-category**, not global:

- `category_types` gains `rollover_enabled INTEGER NOT NULL DEFAULT 1` via a new migration **`004_rollover_toggle.ts`** (`ALTER TABLE category_types ADD COLUMN rollover_enabled INTEGER NOT NULL DEFAULT 1`). The runner applies `up(tx)` in a transaction; SQLite back-fills existing rows with the default. No precedent needed — the runner is generic over `CREATE`/`ALTER`.
- When enabled: `available = allocated + rolledOver − spent` (cumulative formula, budget-row-gated).
- When disabled: `available = allocated − spent` (current month only).
- This column is **sync-safe**: it is a per-category stored fact, merges trivially.
- Default ON matches the dominant envelope-budgeting expectation.

### Where it lives

- New repo function `getRolledOver(db, typeId, month): Promise<number>` — cumulative prior `(allocated − spent)` summed only over months with an existing budgets row for that type, inclusive of overspend going negative.
- Extend `BudgetSummary` with `rolled_over: number` and `available: number` (`= allocated + rolledOver − spent` when enabled, else `allocated − spent`).
- Keep existing `remaining` (`= allocated − spent`) for back-compat with current UI; `available` is the roll-over-aware value the UI should display going forward.
- `getBudgetsForMonth()` resolves the per-category `rollover_enabled` flag and returns `available` accordingly.

### UI

Budget view surfaces `available` (roll-over-aware) instead of (or alongside) `remaining`. A muted annotation (e.g., "+12k rolled over") on categories where `rolled_over !== 0`. Category settings expose the `rollover_enabled` toggle. No global setting.

### Testing

Pure-function tests against `createTestDb()` (real in-memory SQLite). Seed allocations and transactions across 3 months, assert `available` for month 3 reflects months 1–2 surplus/deficit. Cover:
- Surplus rolls forward (under-spend increases next month's available).
- Deficit reduces next month's available (over-spend decreases it).
- Zero-activity month preserves the prior cumulative balance.
- First-ever month has `rolled_over = 0`.
- **Budget-row gating:** spending in a month with no budget row does not affect roll-over (no phantom negative). Roll-over begins only from the first budgeted month.
- **Prior-month refund** reduces that month's spent and thus increases the rolled-forward balance (asserts the refund-aware SQL matters).
- `rollover_enabled = 0` isolates the month (no cumulative contribution).
- **Toggle off mid-history then on again:** the cumulative resumes across the gap (months while disabled are still summed, since the toggle gates *display*, not history) — assert this is intentional or revise.

## Release v0.1.1b — CSV Import Mapping-Profile Data Model

### Concept

Storage shape only — no UI, no importer. Lock in how "mapping profiles" (CSV column → Notchy field assignments) persist, so v0.3 builds the importer UI on a stable base. Cheapest of the three deliverables.

### Storage

One key in the existing `app_meta` table, accessed via existing `getMeta()` / `setMeta()` helpers:

```
key:   'csv_import_profiles'
value: JSON.stringify({ profiles: CsvImportProfile[]; version: 1 })
```

Single key, JSON blob — matches how `integrity_warnings` is already stored. The `version` field supports forward migration if the shape evolves; it stays `1` until something breaks. Profiles are user-created, machine-local, never synced (convenience, not data).

### Shape

```typescript
interface CsvImportProfile {
  id: string;            // ULID
  name: string;          // user label, e.g. "Vietcombank checking"
  delimiter: ',' | ';' | '\t' | '|';
  hasHeader: boolean;
  encoding: 'utf-8' | 'utf-8-bom' | 'windows-1252';  // bank CSV encoding — a profile property, not a parser concern
  mappings: {            // 0-based column index → field
    date?: number;
    amount?: number;
    payee?: number;
    description?: number;
  };
  defaults: {
    account_id: string;  // required — one import targets one account
    kind: 'expense' | 'income' | 'refund';
    tag_id?: string;     // optional fixed tag applied to whole import
  };
}
```

### Key decisions

- **`defaults.account_id` required.** A CSV row does not carry the internal account ID; the profile declares "import into *this* account." This matches ADVICE.md's mapping-profile intent and the v0.3 use case (a bank statement = one account at a time).
- **`tag_id` is a literal default, not a column index.** v0.3 may add column-index tag mapping, but that is deferred until the importer UI exists. Keep the shape minimal now.
- **`encoding` included now** (`utf-8` | `utf-8-bom` | `windows-1252`). Bank CSV encoding is a top source of import pain and is a *profile* property (which bank → which encoding), not a parser concern like `dateFormat`. Cheap to add now, annoying to retrofit across every existing profile later.
- **Deliberately omitted:** `dateFormat`, `amountSign`, column-index `tag_id`. These are importer/parser concerns; v0.3 adds whichever the real bank formats demand. Adding fields later is a non-breaking JSON addition under the same key.
- **No duplicate-detection config.** Dedup is the real hard part of import (users re-import the same statement); deferred to v0.3 where the importer actually runs.

### What ships

1. The `CsvImportProfile` interface + a `csvImportProfiles.ts` repo (`getProfiles()`, `saveProfile()`, `deleteProfile()`) using `getMeta` / `setMeta`. No migration — uses existing `app_meta`.
2. Unit tests: round-trip save/load, JSON shape, `version` field, missing-field tolerance for forward compatibility.

## Release v0.1.2 — Tray Quick-Capture (Desktop)

### Concept

A desktop-only fast path to log a transaction without opening/focusing the main window. Click the tray icon (or press the global shortcut `Ctrl+Shift+N` / `Cmd+Shift+N` on macOS) → a small "quick-add" window appears → type, hit Enter → window hides, transaction saved, main DB updated live.

### Web posture

Tray icons and global OS shortcuts are desktop-only; a browser tab cannot register either. The *concept* of low-friction capture survives on web as an in-page quick-add (e.g., `/` focus shortcut or always-visible dashboard input) — designed when v0.2's web architecture is concrete, **not** in this release. v0.1.2 ships desktop-only.

### Components

1. **Tauri tray plugin** (`tauri-plugin-tray`). Adds: `Cargo.toml` dependency, registration in `src-tauri/src/lib.rs` `setup()`, a tray-capable icon (reuse existing bundle icons — Tauri v2 needs a format it accepts), and a capability entry granting tray permissions (`default.json` currently has none). Tray menu: *Quick Add*, *Show Notchy*, *Quit*. Clicking the icon or *Quick Add* opens the quick-add window.
2. **Global shortcut plugin** (`tauri-plugin-global-shortcut`). Adds: `Cargo.toml` dependency, registration in `lib.rs` `setup()` (the shortcut is **Rust-initiated**, not JS — it cannot be registered from the Svelte layer), and a `global-shortcut:allow-register` capability entry. Binds `Ctrl+Shift+N` (Windows/Linux), `Cmd+Shift+N` (macOS). Must handle registration conflicts (another app owns the combo) gracefully — log and continue, do not crash.
3. **Quick-add window** — a second Tauri window (small, ~360×200, minimal chrome), separate from the main 1200×800 window. Renders a SvelteKit route (e.g., `/quick-add`). Auto-focuses the input on show. Dismisses on submit or `Escape`. (Hide-on-blur is deferred — default behavior only; revisit if it feels wrong in use.)
4. **Single-field input parsing** — the design's core. The existing `parseAmount(locale, currency)` (in `src/lib/utils/number_parse.ts`) is a **pure number evaluator**: it rejects non-numeric characters and throws on results ≤ 0. It has no concept of "leading `+` means income" and cannot ingest a payee string. Therefore quick-add uses a dedicated **tokenizer** in a new module (e.g., `src/lib/utils/quick_parse.ts`) that produces `{ kind, amount, payee }`, then hands only the numeric token to `parseAmount`. `parseAmount` itself is **not extended** — keeping it pure.
   ```
   Grammar:  [sign] <amount>[suffix] <payee…>
   Algorithm:
     1. Note leading '+'  → kind = income (default kind = expense).
     2. Split first whitespace: numeric token | payee remainder.
     3. Expand trailing 'k' (×1000) / 'm' (×1,000,000) on the numeric token.
     4. Hand the bare numeric string to parseAmount(locale, currency) → amount (throws on invalid).
     5. payee = trimmed remainder (may be empty).

   "50k coffee"   → kind=expense, amount=50000,   payee="coffee"
   "1.2m rent"    → kind=expense, amount=1200000, payee="rent"
   "+50k salary"  → kind=income,  amount=50000,   payee="salary"
   ```
   - `k` = ×1,000; `m` = ×1,000,000. VND-conventional, reduces keystrokes, high value for the target locale.
   - Leading `+` sets kind to income. Absence of a sign defaults to expense (the dominant quick-capture case).
   - **Refunds are intentionally not supported in quick-add.** A refund (`kind: 'refund'`) conceptually links to an original expense via `refund_of_id` (validated against a target by `createTransaction`); that target cannot be expressed in shorthand. Refunds use the full TransactionForm. Quick-add handles only expense and income — two kinds, one sign.
5. **Default account** — a new persisted key `default_quick_account_id` in `app_meta` (via `getMeta`/`setMeta`), with repo helpers `getDefaultQuickAccount()` / `setDefaultQuickAccount()`. **Why a new key, not reuse of `session.lastUsedAccountId`:** each Tauri webview is a separate JS context, so the quick-add window has its own session and cannot see the main window's last-used account — it would fall back to `accounts[0]` every time. A persisted meta key remembers the choice across opens and reboots. Falls back to `accounts[0]` if unset. A settings Select lets the user choose it; optionally auto-update it whenever a transaction is saved in the main window (settings-only at first is acceptable). `tag_id` defaults to null; date defaults to today. The window shows these as static plate text (`CHECKING · TODAY`), not pickers.
6. **Window lifecycle & cross-window refresh** — the quick-add window is created once at startup and shown/hidden on demand (faster than create/destroy). On submit: save the transaction via `createTransaction()`, hide the window, clear the input.
   - **Critical architectural fact:** `getDb()` is a per-JS-context module singleton, and each Tauri webview is a separate JS context. The quick-add window therefore gets its **own** `DatabaseService` instance on the same SQLite file (fine under WAL) and its **own** Svelte store. **The main window's transaction list will NOT reactively update on a tray-window save** — store state does not cross windows.
   - **Designed fix:** after `createTransaction()` succeeds, the quick-add window emits a Tauri event (e.g., `transaction:saved`) via the event plugin; the main window `listen`s and refetches the affected stores (transactions, budgets, roll-overs). Without this, "the main list updates" is false. This is a required component, not a deferral.

### Visual design — Tape

The window is a literal adding-machine tape entry. One printed line: phosphor-amber amount glows above a hairline rule, payee sits below as a sans ledger annotation. The whole window reads as one line on a tape, not a mini form. This leans *literally* into Notchy's stated "Adding Machine" aesthetic (DESIGN.md §6) — more on-brand than the main app, which only references the metaphor.

**Layout (360×200 window):**
- Top bar: `▮` mark (left, phosphor glow) · `ESC` (right, dim plate).
- Amount: IBM Plex Mono, `figures-glow`, ~30px. Empty state shows placeholder "Type an amount…" with a blinking caret (input affordance — not decoration; respects `prefers-reduced-motion`).
- Rule: solid `1px var(--line)` at reduced opacity. **Not a gradient** — matches TopBar/Modal/Sidebar dividers.
- Payee: system sans, `--ledger`, ~15px.
- Status line (plate, mono uppercase 11px): `CHECKING · TODAY` (left) · `SAVE ⏎` (right).

**Kind tinting:** expense (default) and income = phosphor glow. (Refund not supported in quick-add; see parser section.)

**Tokens reused — no new colors:** `--tape` background, `--phosphor`/`--phosphor-bright` amount, `--ledger` payee, `--dim` annotations, `--line` rule. Dark is the native surface.

### Testing

- **Tokenizer unit tests** (pure function, `quick_parse.ts`): `50k coffee`, `1.2m rent`, `+50k salary`, bare number (no payee), missing amount (error path), invalid amount delegated to `parseAmount` (error path), `k`/`m` suffix expansion, locale-specific decimal handling. Assert `parseAmount` is called with the bare numeric token only (no payee, no sign).
- **Repo integration:** saved quick-add transaction appears in `getBudgetsForMonth` spend and in the roll-over calc (verifies the cross-feature interaction).
- **Cross-window event test:** after `createTransaction()` in the quick-add window, the `transaction:saved` event is emitted; the main window's listener refetches and the list reflects the new row. (This is the assertion that guards the "main list updates" claim.)
- **E2E (Playwright, desktop-only):** tray/shortcut opens window, typed shorthand saves a transaction, main list updates. Scope to the desktop path; the web quick-add is a separate v0.2 effort.

## Cross-cutting notes

- **Roll-overs ↔ quick-capture interaction:** a quick-added transaction flows through `createTransaction()` → updates spend → affects `available` in the budget view. No special wiring; both use the same repo layer.
- **Sync safety:** `rollover_enabled` (per-category stored fact) and `csv_import_profiles` (machine-local, never synced) are both merge-safe. Roll-over is a derived value, so it never conflicts on merge — only stored allocations do, and those already exist. **Caveat for v0.3/v0.4:** because roll-over is cumulative-derived, a late-arriving synced transaction from months ago retroactively shifts every subsequent month's `available` on every device — correct for envelope budgeting, but a visible cross-device UI churn the sync design will need to handle (invalidate the derived chain). Not a schema foreclosure; flagged so it isn't rediscovered as a surprise.
- **TDD discipline:** all three deliverables follow Red-Green-Refactor per CLAUDE.md. DB-dependent tests use `createTestDb()` + `runMigrations()`. No mocks of the database.
