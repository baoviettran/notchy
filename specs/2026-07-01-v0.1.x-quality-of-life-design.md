**Date:** 2026-07-01
**Status:** Design — pending implementation plan
**Scope:** Two opportunistic patch releases (`v0.1.1`, `v0.1.2`) landing before the v0.2 web build. These are not on the existing roadmap (v0.2 web → v0.3 sync → …); they are cheap, day-to-day-value wins inserted around it.

## Motivation

The project foundation (testing, data integrity, architecture) is solid at v0.1.0. The next pressure is **daily-use stickiness / reduced input friction** — no hard deadline, but work should make the app more pleasant to actually use. The existing roadmap remains the plan; these items are opportunistic extras that do not displace v0.2.

Three deliverables, split along a natural fault line (in-app logic vs. OS integration):

| Release | Deliverable | Kind |

- **CSV importer UI, parser, duplicate detection, column auto-detection.** All v0.3. This spec locks only the storage shape.
- **Web-build quick-add UI.** Tray/global-shortcut are desktop-only; the web (v0.2) will get its own in-page quick-add (e.g., `/` focus shortcut). Designed when v0.2 is concrete.

## Release v0.1.1a — Budget Roll-overs

### Concept

Unspent money in a category carries into the next month's available budget; overspent categories reduce it (envelope budgeting convention). **Pure derived value — no new stored columns for the calculation itself.** The one schema change is a per-category toggle.

### Formula

Roll-over is the cumulative prior surplus/deficit for a category:

```
rolledOver(typeId, month N) = Σ over all months M < N of [allocated(M) − spent(M)]
                            = (lifetime allocated before N) − (lifetime spent before N)
```

The unrolled (cumulative) form is preferred over a month-by-month chain: no recursion, no chaining, no snapshot, one SQL query per category. Recalculating when a past transaction is edited is **correct behavior** — the money really did carry forward; this matches YNAB and envelope-budgeting expectations. Historical-report freezing is a v0.5 reporting concern, out of scope here.

### Per-category toggle

Most categories benefit from roll-over (car repair, gifts — sinking funds). A few want reset (groceries, dining). The escape hatch is **per-category**, not global:

- `category_types` gains `rollover_enabled INTEGER NOT NULL DEFAULT 1`.
- When enabled: `available = allocated + rolledOver − spent` (cumulative formula).
- When disabled: `available = allocated − spent` (current month only).
- This column is **sync-safe**: it is a per-category stored fact, merges trivially.
- Default ON matches the dominant envelope-budgeting expectation.

### Where it lives

- New repo function `getRolledOver(db, typeId, month): Promise<number>` — cumulative prior `(allocated − spent)`, inclusive of overspend going negative.
- Extend `BudgetSummary` with `rolled_over: number` and `available: number` (`= allocated + rolledOver − spent` when enabled, else `allocated − spent`).
- Keep existing `remaining` (`= allocated − spent`) for back-compat with current UI; `available` is the roll-over-aware value the UI should display going forward.
- `getBudgetsForMonth()` resolves the per-category `rollover_enabled` flag and returns `available` accordingly.

### UI

Budget view surfaces `available` (roll-over-aware) instead of (or alongside) `remaining`. A muted annotation (e.g., "+12k rolled over") on categories where `rolled_over !== 0`. Category settings expose the `rollover_enabled` toggle. No global setting.

### Testing

Pure-function tests against `createTestDb()` (real in-memory SQLite). Seed allocations and transactions across 3 months, assert `available` for month 3 reflects months 1–2 surplus/deficit. Cover:
- Surplus rolls forward (under-spend increases next month's available).
- Deficit reduces next month's available (over-spend decreases it).
- Zero-activity month preserves the prior cumulative balance.
- First-ever month has `rolled_over = 0`.
- `rollover_enabled = 0` isolates the month (no cumulative contribution).

## Release v0.1.1b — CSV Import Mapping-Profile Data Model

### Concept

Storage shape only — no UI, no importer. Lock in how "mapping profiles" (CSV column → Notchy field assignments) persist, so v0.3 builds the importer UI on a stable base. Cheapest of the three deliverables.

### Storage

One key in the existing `app_meta` table, accessed via existing `getMeta()` / `setMeta()` helpers:

```
key:   'csv_import_profiles'
value: JSON.stringify({ profiles: CsvImportProfile[]; version: 1 })
```

Single key, JSON blob — matches how `integrity_warnings` is already stored. The `version` field supports forward migration if the shape evolves; it stays `1` until something breaks. Profiles are user-created, machine-local, never synced (convenience, not data).

### Shape

```typescript
interface CsvImportProfile {
  id: string;            // ULID
  name: string;          // user label, e.g. "Vietcombank checking"
  delimiter: ',' | ';' | '\t' | '|';
  hasHeader: boolean;
  mappings: {            // 0-based column index → field
    date?: number;
    amount?: number;
    payee?: number;
    description?: number;
  };
  defaults: {
    account_id: string;  // required — one import targets one account
    kind: 'expense' | 'income' | 'refund';
    tag_id?: string;     // optional fixed tag applied to whole import
  };
}
```

### Key decisions

- **`defaults.account_id` required.** A CSV row does not carry the internal account ID; the profile declares "import into *this* account." This matches ADVICE.md's mapping-profile intent and the v0.3 use case (a bank statement = one account at a time).
- **`tag_id` is a literal default, not a column index.** v0.3 may add column-index tag mapping, but that is deferred until the importer UI exists. Keep the shape minimal now.
- **Deliberately omitted:** `dateFormat`, `amountSign`, column-index `tag_id`. These are importer/parser concerns; v0.3 adds whichever the real bank formats demand. Adding fields later is a non-breaking JSON addition under the same key.
- **No duplicate-detection config.** Dedup is the real hard part of import (users re-import the same statement); deferred to v0.3 where the importer actually runs.

### What ships

1. The `CsvImportProfile` interface + a `csvImportProfiles.ts` repo (`getProfiles()`, `saveProfile()`, `deleteProfile()`) using `getMeta` / `setMeta`. No migration — uses existing `app_meta`.
2. Unit tests: round-trip save/load, JSON shape, `version` field, missing-field tolerance for forward compatibility.

## Release v0.1.2 — Tray Quick-Capture (Desktop)

### Concept

A desktop-only fast path to log a transaction without opening/focusing the main window. Click the tray icon (or press the global shortcut `Ctrl+Shift+N` / `Cmd+Shift+N` on macOS) → a small "quick-add" window appears → type, hit Enter → window hides, transaction saved, main DB updated live.

### Web posture

Tray icons and global OS shortcuts are desktop-only; a browser tab cannot register either. The *concept* of low-friction capture survives on web as an in-page quick-add (e.g., `/` focus shortcut or always-visible dashboard input) — designed when v0.2's web architecture is concrete, **not** in this release. v0.1.2 ships desktop-only.

### Components

1. **Tauri tray icon** (new, in `src-tauri/`). Tray with menu: *Quick Add*, *Show Notchy*, *Quit*. Clicking the icon or the *Quick Add* menu item opens the quick-add window. Requires a tray-capable icon (reuse existing bundle icons).
2. **Global shortcut** `Ctrl+Shift+N` (Windows/Linux), `Cmd+Shift+N` (macOS), registered at app startup. Triggers the quick-add window regardless of app focus state. Must handle registration conflicts (another app owns the shortcut) gracefully — log and continue, do not crash.
3. **Quick-add window** — a second Tauri window (small, ~360×200, minimal chrome), separate from the main 1200×800 window. Renders a SvelteKit route (e.g., `/quick-add`). Auto-focuses the input on show. Dismisses on submit, on `Escape`, or on loss of focus (configurable; default: dismiss on Escape only, hide-on-blur is a common but sometimes-surprising behavior — decide during implementation).
4. **Single-field input parsing** — the design's core. One text field accepting natural shorthand, reusing existing `parseAmount(locale, currency)`:
   ```
   <amount> <payee>

   "50k coffee"   → amount=50000,  payee="coffee",  kind=expense  (default, no sign)
   "1.2m rent"    → amount=1200000, payee="rent",   kind=expense
   "+50k salary"  → amount=50000,  payee="salary",  kind=income
   ```
   - `k` = ×1,000; `m` = ×1,000,000. VND-conventional, reduces keystrokes, high value for the target locale.
   - Leading `+` sets kind to income. Absence of a sign defaults to expense (the dominant quick-capture case).
   - **Refunds are intentionally not supported in quick-add.** A refund (`kind: 'refund'`) conceptually links to an original expense via `refund_of_id` (validated against a target by `createTransaction`); that target cannot be expressed in shorthand. Refunds use the full TransactionForm. Quick-add handles only expense and income — two kinds, one sign.
5. **Smart defaults** — `account_id` defaults to the user's primary account (definition required at implementation: a new `default_account_id` setting in `app_meta`, falling back to the first account). `tag_id` defaults to null. Date defaults to today. The quick-add window shows these as static plate text (`CHECKING · TODAY`), not pickers — matching the existing TransactionForm quick mode's `Account: {name} · Today` pattern. Refining happens later in the main window.
6. **Window lifecycle** — the quick-add window is created once at startup and shown/hidden on demand (faster than create/destroy). On submit: save the transaction via `createTransaction()`, hide the window, clear the input. The main window's transaction list reflects the new row through existing store reactivity (verify both windows share the same SQLite handle / store instance; if not, an event/refresh mechanism is needed — flag for implementation).

### Visual design — Tape

The window is a literal adding-machine tape entry. One printed line: phosphor-amber amount glows above a hairline rule, payee sits below as a sans ledger annotation. The whole window reads as one line on a tape, not a mini form. This leans *literally* into Notchy's stated "Adding Machine" aesthetic (DESIGN.md §6) — more on-brand than the main app, which only references the metaphor.

**Layout (360×200 window):**
- Top bar: `▮` mark (left, phosphor glow) · `ESC` (right, dim plate).
- Amount: IBM Plex Mono, `figures-glow`, ~30px. Empty state shows placeholder "Type an amount…" with a blinking caret (input affordance — not decoration; respects `prefers-reduced-motion`).
- Rule: solid `1px var(--line)` at reduced opacity. **Not a gradient** — matches TopBar/Modal/Sidebar dividers.
- Payee: system sans, `--ledger`, ~15px.
- Status line (plate, mono uppercase 11px): `CHECKING · TODAY` (left) · `SAVE ⏎` (right).

**Kind tinting:** expense (default) and income = phosphor glow. (Refund not supported in quick-add; see parser section.)

**Tokens reused — no new colors:** `--tape` background, `--phosphor`/`--phosphor-bright` amount, `--ledger` payee, `--dim` annotations, `--line` rule. Dark is the native surface.

### Testing

- **Parser unit tests** (pure function): `50k coffee`, `1.2m rent`, `+50k salary`, bare number, missing payee, invalid amount (error path), `k`/`m` suffix, locale-specific decimal handling via `parseAmount`.
- **Repo integration:** saved quick-add transaction appears in `getBudgetsForMonth` spend and in roll-over calc (verifies the cross-feature interaction).
- **E2E (Playwright, desktop-only):** tray/shortcut opens window, typed shorthand saves a transaction, main list updates. Scope to the desktop path; the web quick-add is a separate v0.2 effort.

## Cross-cutting notes

- **Roll-overs ↔ quick-capture interaction:** a quick-added transaction flows through `createTransaction()` → updates spend → affects `available` in the budget view. No special wiring; both use the same repo layer.
- **Sync safety:** `rollover_enabled` (per-category stored fact) and `csv_import_profiles` (machine-local, never synced) are both merge-safe. Roll-over is a derived value, so it never conflicts on merge — only stored allocations do, and those already exist.
- **TDD discipline:** all three deliverables follow Red-Green-Refactor per CLAUDE.md. DB-dependent tests use `createTestDb()` + `runMigrations()`. No mocks of the database.
